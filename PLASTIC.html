PLASTIC: Green Query Optimization

A proposal to eliminate wasteful computation in commercial databases.

sync_problem

Databases waste significant resources by optimizing nearly every complex query from scratch.

Query optimization is a computationally intensive and costly process for any modern database. The system typically treats each query as a new problem, optimizing it "afresh" with no mechanism to amortize the high upfront cost over prior, similar queries. This default behavior creates a persistent and expensive performance bottleneck.

price_check

This constant re-optimization represents a major, ongoing drain on computational and financial resources.

Wasted Time

  • timerThe overhead of re-optimization directly translates to slower query response times. As workloads become more complex, this bottleneck worsens, impacting application performance and user experience.

Wasted Resources

  • memoryEvery optimization cycle consumes valuable CPU and memory. This brute-force approach inflates the hardware and energy costs required to run the database, directly impacting the operational budget.
document_scanner

Existing plan reuse tools in commercial databases are too limited because they rely on superficial text matching.

Commercial systems like Oracle 9i offer features such as "stored outlines" to reuse plans, but they are fundamentally flawed. Their matching is extremely restrictive, requiring a new query to have a close textual resemblance to a stored one. This misses the vast majority of opportunities, as queries that are structurally identical but textually different are ignored.

smart_toy

PLASTIC introduces "Plan Recycling," a novel technique to intelligently reuse optimization work for structurally similar queries.

PLASTIC is a tool designed to help query optimizers significantly amortize their overheads. Rather than relying on text, we identify deep similarities between queries. By grouping queries into clusters, we can use the high-quality, optimizer-generated plan for a single "representative" to execute all future queries in that cluster. This allows the cost of a high-quality optimization to be spread across thousands of queries.

architecture

Our system characterizes, clusters, and classifies queries to find and reuse the optimal plan template.

Characterize

  • analyticsEach query is processed by a Feature Vector Extractor, which captures its key structural and statistical properties from the system catalogs.

Cluster

  • bubble_chartA Similarity Check module compares the query's vector to existing cluster representatives. If no match is found, the optimizer generates a new plan and the query becomes a new representative.

Execute

  • assignment_turned_inIf a match is found, the stored Plan Template is retrieved and the Plan Generator module uses it to execute the current query, saving the full optimization cost.
